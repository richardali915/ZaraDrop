# ZaraDrop Enterprise Architecture

## Vision
ZaraDrop is an enterprise-grade logistics operating system built for Africa, starting in Abuja and designed to scale to 10,000+ deliveries per day without a rewrite. This architecture builds on the existing monorepo, preserving the current SvelteKit apps, shared packages, and Supabase edge functions while upgrading the platform into a production-ready enterprise system.

## Core Principles
- Reliability first
- Data accuracy and auditability
- Role-driven access control
- Real-time visibility for dispatch and operations
- Modular shared services inside one monorepo
- Strong separation of presentation, logic, and data
- Honest resilience for low-end Android and intermittent networks
- Security, compliance, and fraud defense from the beginning

## Monorepo Strategy

## Shared Services Architecture
### Auth and identity
- Single authentication system powered by Supabase Auth
- All users are represented in `auth.users`
- Extended profile tables for role-specific information
- Roles: `customer`, `rider`, `merchant`, `support_agent`, `operations_manager`, `finance_manager`, `admin`, `super_admin`
- Role-based permissions and dynamic access control through a shared permission table
- Device session tracking and multi-device session revocation

### Notification engine
- Primary delivery: Firebase Cloud Messaging for mobile/web push
- Secondary delivery: Supabase notifications table + email/SMS adapters
- Event templates in a shared notification package
- Persist notification preferences and history

### Shared business logic
- `@zaradrop/lib`: core Supabase helpers, function callers, wallet/checkout abstractions, delivery assignment helper
- `@zaradrop/hooks`: session and auth state, role guards, global stores, retry-safe network wrappers
- `@zaradrop/types`: strong schema contracts for orders, deliveries, settlements, user profiles, and notifications
- `@zaradrop/ui` + `@zaradrop/styles`: single design system with component-level CSS and theme tokens

## Database Architecture
### Design goals
- Single PostgreSQL database with normalized domain schema
- Strong foreign keys, indexes, materialized views for reporting
- Audit trails and immutable event records
- Row Level Security with role-aware access
- Separation of live operational tables and reporting tables

### Core data model
- `users`: shared auth identity
- `roles`, `permissions`, `user_roles`, `role_permissions`
- `customers`, `riders`, `merchants`, `support_agents`, `operations_managers`, `finance_managers`, `admins`
- `addresses`, `locations`, `zones`
- `orders`, `order_items`, `order_statuses`, `deliveries`, `delivery_tracking`
- `wallets`, `wallet_transactions`, `settlements`, `settlement_items`, `commissions`, `service_fees`
- `notifications`, `notification_preferences`
- `support_tickets`, `support_messages`, `ticket_assignments`, `escalations`
- `audit_logs`, `admin_actions`, `fraud_reports`
- `analytics_events`, `payment_logs`, `device_sessions`, `suspensions`

### Real-time and dispatch
- `delivery_tracking` stores GPS/history points for each active delivery
- `order_statuses` and `delivery_status` are normalized status wheels, not loose strings
- `dispatch_assignments` and `rider_shifts` support automated and manual routing
- `ride_capacity` and `zone_availability` surfaces dispatch constraints

### Security and compliance
- RLS policies on all tables, especially payments, wallet, notifications, and orders
- `audit_logs` capture changes generated by edge functions and app actions
- `fraud_reports` support detection of suspicious accounts, delivery anomalies, and payment risk

## Role & Permission System
### Roles
- Customer
- Rider
- Merchant
- Support Agent
- Operations Manager
- Finance Manager
- Admin
- Super Admin

### Permissions
- View own orders, wallet, profile
- Manage customer order lifecycle
- Manage merchant onboarding and products
- Manage rider onboarding and shifts
- Assign and override deliveries
- View and export financial reports
- Manage support queue and SLA escalations
- Audit system configuration and security events

### Access controls
- Role assignments are stored in `user_roles`
- Permission checks happen both at the application layer and in SQL policies
- Super Admin has an explicit bypass scope for emergency operations

## UI Architecture

- Shared guard and route layout for auth state in `hooks/`
- Domain-driven routes:
  - `/dashboard`, `/orders`, `/tracking`, `/profile`, `/support`
  - `/rider/availability`, `/rider/jobs`, `/rider/earnings`
  - `/merchant/orders`, `/merchant/store`, `/merchant/settlements`
  - `/admin/approvals`, `/admin/audit`, `/admin/finance`
  - `/operations/dispatch`, `/operations/heatmap`, `/operations/incidents`
  - `/support/tickets`, `/support/chat`, `/support/escalations`

## Dispatch Architecture
- Delivery assignment engine lives in Supabase edge functions and shared service helpers
- Order matching informed by:
  - rider availability and status
  - rider distance to pickup zone
  - rider rating and performance tier
  - delivery urgency and merchant SLA
  - bag size, vehicle capacity, and route load
- Real-time route recalculation via background function or dispatch worker
- Delivery in-flight state transitions captured via `delivery_tracking`
- Failure modes: reassign, auto-escalate, rush re-dispatch

## Rider Architecture
- Multi-step rider onboarding
  - profile submit
  - identity verification
  - vehicle registration
  - document upload
  - approval workflow
- Status states: `pending`, `approved`, `rejected`, `suspended`
- Shift model for availability and peak-hour scheduling
- Offline mode support with local queueing and recovery on reconnect
- Rider incident reporting and SOS tool support
- Earnings and settlement history in the rider dashboard

## Merchant Architecture
- Business onboarding and approval workflow
- Store and product management
- Order acceptance and fulfillment workflow
- Merchant performance metrics and commission visibility
- Settlement reporting and invoice exports
- Business settings and payout configuration

## Customer Architecture
- Authentication and profile management
- Saved addresses and delivery preferences
- Wallet funding and transaction history
- Order creation and tracking with live route map
- Refund and complaint flow
- Ratings and review capture after delivery
- Support ticket creation from order pages

## Admin Architecture
- Command center for approvals and compliance
- Unified user management for customers, riders, merchants
- Finance and settlement controls
- Fraud and incident investigation tools
- Audit trails and system settings controls
- Notification broadcasting and support queue overview

## Support Architecture
- Ticketing queue and agent assignment
- Chat and escalation channels
- SLA monitoring and priority classification
- Internal notes and ticket history
- Analytics for response time and resolution rate

## Finance Architecture
- Commission engine and fees
- Settlement automation for riders and merchants
- Transaction auditing and payout reconciliation
- Revenue dashboards and export tools
- Integration points for payment gateway and bank transfers

## Notification Architecture
- Event-driven push notifications via FCM
- Persistent notifications in `notifications` table
- Preference controls by channel and event type
- Notification templates with merge fields
- Audit-safe notification send logs

## Security Architecture
- Supabase Auth for authentication, enhanced with MFA in future phases
- Role-based authorization enforced in UI and SQL policies
- Audit logs for all administrative actions
- Input validation and rate-limiting in edge functions
- Device session tracking and session revocation
- Fraud detection rules and account suspension workflow

## Monitoring Architecture
- Metrics collection in `analytics_events`
- Delivery health dashboards and alerts for failed/late deliveries
- Audit monitoring for admin actions, policy changes, user suspensions
- Realtime error tracking in front-end and server logs

## Deployment Architecture
- Use Supabase for database, auth, storage, realtime, edge functions
- Deploy SvelteKit apps to Vercel/Netlify or Supabase Edge if needed
- Use GitHub Actions / CI pipelines for build, test, and deployment
- Manage environment configurations per app using `.env` and secrets

## Testing Strategy
- Shared unit tests for `packages/types`, `packages/lib`, `packages/hooks`
- Integration tests for major app workflows
- End-to-end tests for customer, rider, merchant, and admin scenarios
- Migration tests for Supabase schema changes
- Load/performance validation for core delivery and wallet flows

## Scaling Strategy
- Start small with strong normalization and targeted indexes
- Use Supabase realtime for live updates; scale event load with filtered subscriptions
- Optimize queries with composite indexes and materialized reporting views
- Use caching for non-critical dashboard aggregates
- Load test the dispatch and payment flows before 3,000+ deliveries/day

## Production Launch Plan
1. Harden authentication and RBAC with RLS policies.
2. Build first end-to-end workflows: customer order -> merchant acceptance -> rider assignment -> delivery completion.
3. Add rider and merchant onboarding approval workflows.
4. Create the admin approval center and operations dispatch room.
5. Implement wallet, settlements, and notifications.
6. Deploy to staging, run pilot deliveries in Abuja, gather operational metrics.
7. Harden monitoring, fraud controls, incident escalation.
8. Launch production with a safe rollout plan and support center.
